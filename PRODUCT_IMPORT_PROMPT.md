# Product Import System — Implementation Prompt

## Overview

Build a comprehensive bulk product import system for the admin dashboard that allows administrators to upload products via a CSV file bundled with real image files inside a ZIP archive. The system should also support a secondary "two-part upload" method where the CSV and images are selected separately in the same form.

**Tech Stack:** Next.js 15 (App Router), React 19, TypeScript, Supabase (PostgreSQL + Storage), Tailwind CSS, deployed on Vercel.

**Existing Infrastructure:**
- Supabase Storage bucket named `products` already exists with public read access and admin-only write policies.
- Product images are stored in the `product_images` table with columns: `id`, `product_id`, `url`, `alt_text`, `position`, `width`, `height`, `created_at`.
- Products table fields (only these are used by the existing ProductForm): `name`, `slug` (auto-generated), `sku` (auto-generated, format `SLI-XXXX-XXXX`), `description`, `category_id`, `price`, `compare_at_price`, `quantity`, `moq`, `status` (enum: `active`/`draft`/`archived`), `featured`, `seo_title`, `seo_description`, `tags` (text array), `metadata` (jsonb containing `low_stock_threshold` and `preorder_shipping`).
- Product variants table fields (as used by ProductForm): `product_id`, `name` (the size value), `sku`, `price`, `quantity`, `option1` (size), `option2` (color name), `metadata` (jsonb containing `color_hex`).
- Categories table has: `id`, `name`, `slug`, `parent_id`, `image_url`, `position`, `status` (enum: `active`/`inactive`).
- Existing image upload logic in `components/admin/ProductForm.tsx` uploads to Supabase Storage `products` bucket and stores the public URL.
- Admin Supabase client is in `lib/supabase-admin.ts` (service role key, bypasses RLS).
- Client Supabase client is in `lib/supabase.ts`.
- The ProductForm variant system uses a **Color × Size grid** — users pick from preset colors (with hex values) and preset sizes, and variants are auto-generated as every combination. Preset colors: Black, White, Red, Blue, Navy, Green, Yellow, Pink, Purple, Orange, Gray, Brown, Beige, Maroon, Teal, Cream, Gold, Silver. Preset sizes: XS, S, M, L, XL, XXL, 3XL.

## ZIP Archive Format

The user uploads a single `.zip` file with the following structure:

```
product-import.zip
├── products.csv
└── images/
    ├── product-image-1.jpg
    ├── product-image-2.jpg
    ├── product-image-3.png
    └── ...
```

### CSV Column Specification

These columns match exactly what the existing `ProductForm.tsx` submits to the database. Headers are case-insensitive and trimmed.

**Required Columns:**
| Column | Description |
|--------|-------------|
| `name` | Product name |
| `price` | Product price in GH₵ (numeric) |

**Optional Columns:**
| Column | Description |
|--------|-------------|
| `description` | Product description (plain text, max ~500 chars to match the form's limit) |
| `category` | Category name (matched against existing categories by name, case-insensitive). Must match an active category in the database. |
| `compare_at_price` | Original/compare price for showing discounts (numeric). Must be higher than `price`. |
| `quantity` | Stock quantity (integer, defaults to 0). If variants are provided, this is ignored — total stock is the sum of all variant stocks. |
| `moq` | Minimum order quantity (integer, defaults to 1) |
| `status` | Product status: `Active`, `Draft`, `Archived` (defaults to `Draft`) |
| `featured` | Whether product is featured on homepage: `true`/`false` (defaults to `false`) |
| `seo_title` | SEO page title (60 chars recommended) |
| `seo_description` | SEO meta description (160 chars recommended) |
| `keywords` | Comma-separated keywords/tags (e.g., `"earbuds,bluetooth,wireless"`). These become the product's `tags` array. |
| `low_stock_threshold` | Get notified when stock falls below this number (integer, defaults to 5). Stored in `metadata.low_stock_threshold`. |
| `preorder_shipping` | Pre-order / estimated shipping text (e.g., `"Ships in 14 days"`). Leave empty if product ships immediately. Stored in `metadata.preorder_shipping`. |
| `images` | Semicolon-separated image filenames referencing files in the `images/` folder (e.g., `"front.jpg;side.jpg"`). First image becomes the primary image. |

**Note:** The following fields are NOT used because the current ProductForm does not collect them: `brand`, `vendor`, `short_description`, `weight`, `weight_unit`, `cost_per_item`, `barcode`. If these are added to the form in the future, the import system should be updated to support them.

**Note:** `sku` is auto-generated by the system using the format `SLI-XXXX-XXXX` (matching the `generateSku()` function in ProductForm). Do not include an SKU column — every imported product gets a unique auto-generated SKU. Similarly, `slug` is auto-generated from the product name.

**Variant Columns (optional — Color × Size grid system):**

The existing ProductForm uses a Color × Size grid to create variants. Each variant is a combination of a color and/or a size.

| Column | Description |
|--------|-------------|
| `variant_color` | Color name (e.g., `"Black"`, `"Red"`, `"Navy"`). Must match one of the preset colors or be treated as a custom color. |
| `variant_color_hex` | Hex color code (e.g., `"#000000"`, `"#EF4444"`). Stored in variant `metadata.color_hex`. If omitted for a known preset color, the system should auto-fill the hex. |
| `variant_size` | Size value (e.g., `"S"`, `"M"`, `"L"`, `"XL"`, `"42"`, `"100ml"`). Stored as variant `name` and `option1`. |
| `variant_price` | Variant-specific price (defaults to the product's main `price` if omitted) |
| `variant_stock` | Variant stock quantity (integer, defaults to 0) |

**Preset color mapping (auto-fill hex if `variant_color_hex` is omitted):**
Black=#000000, White=#FFFFFF, Red=#EF4444, Blue=#3B82F6, Navy=#1E3A5F, Green=#22C55E, Yellow=#EAB308, Pink=#EC4899, Purple=#A855F7, Orange=#F97316, Gray=#6B7280, Brown=#92400E, Beige=#D2B48C, Maroon=#800000, Teal=#14B8A6, Cream=#FFFDD0, Gold=#D4AF37, Silver=#C0C0C0

**Preset sizes:** XS, S, M, L, XL, XXL, 3XL (custom sizes like "42", "100ml" are also allowed)

> **How variants work in the CSV:** If a product has multiple variants, repeat the product on multiple rows with the same `name` but different `variant_*` values. The system groups rows by `name` and creates one product with multiple variants. Each row represents one Color × Size combination.
>
> **How variants are saved to the database** (matching ProductForm behavior):
> - `product_variants.name` = the size value (or the color if no size)
> - `product_variants.option1` = the size value
> - `product_variants.option2` = the color name
> - `product_variants.price` = variant price
> - `product_variants.quantity` = variant stock
> - `product_variants.metadata` = `{ "color_hex": "#XXXXXX" }` if color is provided
> - The parent product's `quantity` = sum of all variant stocks

### 2. API Route

Create the API route at `app/api/admin/products/import/route.ts`.

**Endpoint:** `POST /api/admin/products/import`

**Request:** `multipart/form-data` with either:
- A single `zipFile` field containing the `.zip` file, OR
- A `csvFile` field + multiple `images` fields

**Authentication:** Verify the user is authenticated and has admin/manager role. Return 401/403 if not.

**Processing Pipeline:**

1. **Extract/Parse Phase:**
   - If ZIP: extract using `jszip` library. Find the CSV file (any `.csv` file at root level of the archive). Collect all image files from any `images/` directory (or root level).
   - If separate files: use the CSV file and image files directly.
   - Build a `Map<string, Buffer>` mapping lowercase filenames to image file buffers.

2. **Validation Phase:**
   - Parse CSV using a library like `papaparse` (or `csv-parse`).
   - Validate headers — check that required columns (`name`, `price`) exist.
   - Validate each row:
     - `name` is present and non-empty
     - `price` is a valid positive number
     - `compare_at_price` (if provided) is a valid number higher than `price`
     - `quantity` (if provided) is a non-negative integer
     - `moq` (if provided) is a positive integer >= 1
     - `status` (if provided) is one of: `Active`, `Draft`, `Archived`
     - `featured` (if provided) is `true` or `false`
     - `low_stock_threshold` (if provided) is a non-negative integer
     - `images` (if provided) — check that every referenced filename exists in the images map
     - `category` (if provided) — check that the category exists in the database (case-insensitive match on name against active categories)
     - `variant_color_hex` (if provided) is a valid hex color (e.g., `#000000`)
     - `variant_stock` (if provided) is a non-negative integer
   - Collect all validation errors with row numbers but don't stop — validate everything first.
   - If there are critical errors (missing required fields), mark those rows as failed but continue processing valid rows.

3. **Image Upload Phase:**
   - For each unique image filename referenced in valid product rows:
     - Generate a unique storage path: `imports/${timestamp}/${original-filename}` to avoid collisions
     - Upload to Supabase Storage `products` bucket using the admin client
     - Get the public URL
     - Store in a `Map<string, string>` mapping original filename to public URL
   - Track upload progress and report it back to the client

4. **Product Creation Phase:**
   - Group rows by `name` to identify products vs. variant rows (SKU is not in the CSV — it's auto-generated).
   - For each product group:
     - Generate a `slug` from the product name: `name.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/(^-|-$)+/g, '')` — ensure uniqueness by appending `-1`, `-2`, etc. if collisions occur
     - Generate a `sku` using the same format as `ProductForm.tsx`: `SLI-${timestamp.toString(36).toUpperCase().slice(-4)}-${random.toString(36).substring(2,6).toUpperCase()}`
     - Look up `category_id` from the category name (case-insensitive match against active categories)
     - Parse `keywords` from comma-separated string into a `tags` text array
     - Build `metadata` object: `{ low_stock_threshold: <number>, preorder_shipping: <string|null> }`
     - If variants exist, set `quantity` = sum of all variant stocks
     - Insert into `products` table
     - Insert image records into `product_images` table with: `product_id`, `url` (the uploaded Supabase public URL), `position` (index), `alt_text` (product name)
     - If variant rows exist, insert into `product_variants` table with: `product_id`, `name` (size or color), `option1` (size), `option2` (color), `price`, `quantity`, `metadata` (`{ color_hex }` if provided)
   - Use database transactions — each product (with its images and variants) should be inserted atomically
   - If a single product fails, log the error and continue with the next product (don't fail the entire import)

5. **Response:**
   - Return a JSON response with the full results:
   ```json
   {
     "success": true,
     "summary": {
       "totalRows": 500,
       "productsCreated": 485,
       "variantsCreated": 120,
       "imagesUploaded": 970,
       "errors": 15,
       "warnings": 5,
       "skipped": 0,
       "duration": "45s"
     },
     "errors": [
       { "row": 15, "field": "name", "message": "Name is required" },
       { "row": 23, "field": "images", "message": "File 'missing.jpg' not found in archive" }
     ],
     "warnings": [
       { "row": 44, "field": "category", "message": "Category 'Unknown' not found. Product created without category." }
     ]
   }
   ```

### 3. Streaming Progress (Important for Large Imports)

Since imports of thousands of products with images can take several minutes, implement progress streaming:

**Option A — Server-Sent Events (SSE):**
- Change the API route to return a `ReadableStream` with `Content-Type: text/event-stream`.
- Send progress events as the import proceeds:
  ```
  event: progress
  data: {"phase":"validating","current":50,"total":500,"message":"Validating row 50 of 500..."}

  event: progress
  data: {"phase":"uploading_images","current":100,"total":970,"message":"Uploading image 100 of 970..."}

  event: product
  data: {"row":1,"name":"Blue Sneaker","status":"success"}

  event: product
  data: {"row":2,"name":"Red T-Shirt","status":"error","error":"Invalid price value"}

  event: complete
  data: {"summary":{...}}
  ```
- On the frontend, use `EventSource` or `fetch` with a `ReadableStream` reader to consume events and update the UI in real-time.

**Option B — Polling (simpler alternative):**
- The initial POST returns an `importId`.
- Store import progress in the database (create an `import_jobs` table) or in-memory.
- Frontend polls `GET /api/admin/products/import/[importId]/status` every 2 seconds.
- Less ideal but simpler to implement.

**Recommend Option A (SSE)** for a better user experience.

### 5. Duplicate Handling

Since SKUs are auto-generated (not provided in the CSV), duplicate detection should be based on product `name`:
- **Default behavior:** If a product with the same `name` already exists in the database, skip it and log a warning.
- **Optional:** Add a checkbox in the UI — "Update existing products if name matches" — which, when enabled, will upsert instead of skip. If updating, merge new images with existing ones (don't delete old images).

### 6. Error Handling & Edge Cases

- **Empty CSV:** Show error "CSV file is empty or contains only headers."
- **No images folder in ZIP:** Allow it — products will be created without images (log warnings for rows that reference images).
- **Unsupported image formats:** Skip the image and log a warning. Only allow: `.jpg`, `.jpeg`, `.png`, `.webp`, `.gif`.
- **Very large images:** If an image exceeds 10MB, skip it and log a warning. Optionally, compress/resize images server-side before uploading (using `sharp` library).
- **Malformed CSV:** If the CSV can't be parsed at all, return an error immediately.
- **ZIP bomb protection:** Limit total extracted size to 1GB. If exceeded, abort and return error.
- **Duplicate image filenames in CSV:** Upload the image once, reuse the URL for all products referencing it.
- **Special characters in filenames:** Normalize filenames (lowercase, replace spaces with hyphens) when uploading to storage.
- **Network interruption:** If the upload is interrupted, any already-uploaded images remain in storage (orphaned). This is acceptable — a cleanup job can be added later.

### 8. Database Considerations

- The `slug` field on `products` has a unique constraint. Generate slugs from the product name using `name.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/(^-|-$)+/g, '')` and append `-1`, `-2`, etc. if collisions occur.
- The `sku` field on `products` has a unique constraint. Auto-generate unique SKUs using the format `SLI-${Date.now().toString(36).toUpperCase().slice(-4)}-${Math.random().toString(36).substring(2,6).toUpperCase()}`. Do NOT accept SKU from CSV.
- Use Supabase admin client (`lib/supabase-admin.ts`) for all database operations during import to bypass RLS.
- Wrap each product + its images + its variants in a single database transaction.
- For category matching, query all categories once at the start and build a lookup map (lowercase name → id) to avoid repeated queries. Only match against categories with `status = 'active'`.
- For variant color presets, build a lookup map of color name → hex so the system auto-fills `variant_color_hex` when a known preset color name is provided without a hex value.

### 10. Security Requirements

- Verify admin authentication on the API route before processing anything.
- Validate file types server-side (don't trust client-side validation alone).
- Sanitize all CSV values before database insertion (especially `description` which allows HTML — run it through a sanitizer like `DOMPurify` or `sanitize-html`).
- Set maximum file size limits on the API route (500MB for ZIP, 10MB for CSV, 10MB per image).
- Rate limit the import endpoint to prevent abuse (e.g., max 5 imports per hour per user).
- Log the import action in the audit log with: who initiated it, when, how many products were imported, and the result.
